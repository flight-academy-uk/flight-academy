## Context

[ADR-005 §E](ADR-005-workspace-layout.md) and [ADR-006 §A](ADR-006-api-contract.md)
specify the emitted OpenAPI spec is written to `docs/api/openapi.yaml`. Wiring
utoipa's `to_yaml()` in the walking-skeleton HTTP-foundation commit surfaced
the cost: it requires utoipa's `yaml` feature, which transitively pulls
`serde_norway` and **`unsafe-libyaml-norway`** — a C-to-Rust port of `libyaml`
carrying thousands of lines of mechanical unsafe.

The cost is supply-chain, not runtime. The `emit-spec` subcommand runs in
CI/dev to produce the committed artifact; no incoming HTTP request reaches
the YAML path. But the crate is still compiled and linked into every build,
and [ADR-017](ADR-017-outbound-http-ssrf.md)'s planned `cargo deny`
configuration plus [ADR-016 §C/§E](ADR-016-compliance-baseline.md)'s OWASP
ASVS L2 and Cyber Essentials Plus alignment will both need a stance on every
transitive unsafe dep. Allow-listing one carries an ongoing re-evaluation
cost that does not exist if the dep was never pulled.

`serde_json` is already required across the stack — every JSON response
body, every `application/problem+json` error envelope
([ADR-006 §F](ADR-006-api-contract.md)), every audit-event payload
([ADR-004 §D](ADR-004-defence-in-depth.md)) goes through it. Its `unsafe`
surface is small, well-audited, and bounded to UTF-8 fast paths and
integer/float formatting — not mechanical transliteration of C.

`utoipa::openapi::OpenApi` implements `serde::Serialize`; the serializer
is swappable per call site. If a pure-safe-Rust YAML crate matures or a
bespoke safe emitter is written, the swap-back is one line in
`apps/api/src/main.rs`.

Forces: supply-chain hygiene
([ADR-016](ADR-016-compliance-baseline.md), [ADR-017](ADR-017-outbound-http-ssrf.md));
restraint on transitive deps
([CODE_OF_ETHICS.md](../../CODE_OF_ETHICS.md) instruments 35–36); honesty
about which threat model is load-bearing (instrument 28) — runtime YAML
parsing is not a risk this codebase has; build-time supply-chain exposure
is; the swap-back path is preserved without claiming permanence
(instrument 24).

## Decision

**The emitted OpenAPI document is JSON at `docs/api/openapi.json`, produced
by `serde_json::to_string_pretty` via utoipa's `to_pretty_json()`. The
utoipa `yaml` feature is not enabled. ADR-005 §E and ADR-006 §A read
through this ADR — the committed artifact is `docs/api/openapi.json`,
not `openapi.yaml`.**

The OpenAPI specification accepts either JSON or YAML interchangeably; tooling
consumers may convert one to the other (e.g. `yq -P . docs/api/openapi.json`)
without us shipping both. The contract is the spec content, not its
serialization format.

### Implementation shape

- `apps/api/Cargo.toml` declares `utoipa` without the `yaml` feature.
- `apps/api/src/main.rs`'s `emit-spec` subcommand calls
  `flight_academy_api::openapi().to_pretty_json()`, writing to stdout.
- The committed artifact at `docs/api/openapi.json` is regenerated by CI
  per [ADR-006 §A](ADR-006-api-contract.md) and diffed against the emitted
  output.

## Consequences

**Positive.** ~800–1500 fewer `unsafe` blocks in the dep tree
(`unsafe-libyaml-norway` and its supporting infrastructure removed entirely);
no transitive C-port to audit; pre-empts the `cargo deny` allow-list question
for the YAML lineage; smaller release binary; faster build. The decision uses
a serializer the rest of the stack already requires, so the overall
supply-chain surface does not grow.

**Negative.** JSON diffs are slightly noisier than YAML (curly braces,
quoted keys, no implicit anchors). The developer-ergonomic loss for
human-reading the committed spec is real but small. OpenAPI tooling consumers
who specifically want YAML must convert.

**Neutral.** Industry convention is split — Stripe ships JSON, Kubernetes
ships YAML, GitHub publishes both. No strong external signal either way.
utoipa supports both formats natively; the choice is per-call-site.

## Alternatives considered

- **Keep YAML, allow-list the transitive `unsafe`.** Defensible — the unsafe
  is a vetted `libyaml` port, not novel. Rejected because the
  cost-of-keeping is recurring (every audit pass, every contributor, every
  `cargo deny` review re-asks the question) while the cost-of-switching is
  one line in `main.rs`, and `serde_json` is already required elsewhere.
  Symmetry test: if we were emitting JSON first and someone proposed adding
  YAML on top, the answer would be "what does it buy?"
- **Emit both formats.** Worst of both worlds — keeps the YAML dep, adds CI
  surface to verify both stay in lock-step, doubles the diff-review burden.
  Rejected.
- **Custom safe YAML emitter for the OpenAPI subset.** Realistic — emit-only
  (no parsing) for the bounded shape OpenAPI uses is on the order of
  200–400 lines of safe Rust. Rejected at v1 as over-engineering; named here
  as a follow-on option if a concrete consumer pain emerges. Would live in
  its own crate (`flight-academy-yaml-emit` or similar) and warrant its own
  ADR.
- **Post-process JSON → YAML at the CI layer** (a Node `yaml`-package step,
  a `yq` step). Workable for distribution if specific tooling consumers
  need YAML; not chosen at v1 because nobody needs it yet and shipping JSON
  only is the simpler-and-honest position.
- **Defer the decision — keep YAML for the walking-skeleton commit, revisit
  later.** Rejected because deferring means we add the dep, write code
  against it, then have to migrate code and write an ADR later. Cheaper to
  decide now while the implementation is one commit deep.

## Notes

This ADR makes one decision — JSON over YAML for the emitted spec. It is
**not** a general policy on transitive `unsafe`. A broader supply-chain ADR
would name evaluation criteria (audit record, novelty of unsafe vs port,
sandboxing options, allow-list rules) and would be authored when the
`cargo deny` configuration lands. Until then, the discipline is
case-by-case under the principle here: *if a serializer the codebase already
requires will do the job, do not add a second one with a heavier unsafe
footprint.*

The runtime-vs-build-time framing is important enough to record explicitly:
this ADR does not justify itself by a runtime risk that does not exist.
The risk is supply-chain (build-host compromise, audit cost, recurring
re-evaluation cost), and removing it is negligible-cost because the
alternative serializer is already in the tree.
